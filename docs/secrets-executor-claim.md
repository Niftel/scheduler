# Secrets-service executor claim protocol

## Purpose

Bind a dispatched execution run to the exact executor workload that may resolve
its credential. The scheduler cannot create this binding at initial dispatch:
the durable NATS consumer selects an executor only after the request is
published.

## Security requirements

- Executor identity comes only from a verified TLS 1.3 client certificate URI
  SAN: `spiffe://<trust-domain>/workload/praetor-executor/<instance>`.
- The shared ingestion bearer token is not accepted by the claim endpoint and
  no identity header is trusted.
- A run can be claimed by exactly one executor identity. Replays by that same
  identity are idempotent; a different identity receives a conflict.
- The scheduler reads the external credential UUID/version through the run's
  snapshotted local credential reference. Neither is supplied by the executor.
- The scheduler registers the secrets-service binding before acknowledging a
  successful claim. Secret values never enter this request or the scheduler.
- Claims are allowed only for non-terminal runs whose outbox launch has been
  committed.

## Endpoint

The scheduler exposes a dedicated mTLS listener, separate from metrics:

`POST /internal/v1/execution-runs/{run_id}/claim`

The listener is enabled only when `PRAETOR_SECRETS_URL` is set. Enabling it
requires all of the following settings; scheduler startup fails if any are
missing or invalid:

| Setting | Purpose |
| --- | --- |
| `PRAETOR_SECRETS_URL` | HTTPS URL of the secrets service |
| `PRAETOR_SECRETS_CA_FILE` | CA used to verify the secrets service |
| `PRAETOR_SECRETS_CERT_FILE` | Scheduler client certificate for the secrets service |
| `PRAETOR_SECRETS_KEY_FILE` | Scheduler client private-key file |
| `PRAETOR_SECRETS_TRUST_DOMAIN` | Expected SPIFFE trust domain for workload identities |
| `PRAETOR_CLAIM_ADDR` | Claim listener address; defaults to `:8443` |
| `PRAETOR_CLAIM_CERT_FILE` | Claim listener server certificate |
| `PRAETOR_CLAIM_KEY_FILE` | Claim listener server private-key file |
| `PRAETOR_CLAIM_CLIENT_CA_FILE` | CA used to verify executor client certificates |

The outbound scheduler identity and inbound listener identity deliberately use
separate certificate settings. This permits independent issuance and rotation,
and prevents trusting the secrets-service server CA as an executor issuer by
accident. Private keys are accepted only as file paths.

Request body:

```json
{"dispatch_id":"8c754858-d2fc-42cb-9c60-9cefc2508cef"}
```

`dispatch_id` is generated by the scheduler and included in the execution
request. It prevents a stale redelivery from claiming a newer dispatch of the
same logical job.

Success returns `200` with no credential identifiers. The executor may then
call the secrets service's run-scoped resolve endpoint using its own mTLS
identity.

## Atomic state transition

Inside a database transaction, the scheduler:

1. locks the execution run;
2. verifies the dispatch ID, runnable state, and existing claim;
3. reads `credentials.secrets_service_id` and
   `credentials.secrets_service_version` from the snapshotted credential;
4. calls `RegisterBinding` with the certificate-derived executor identity;
5. records the executor identity and binding timestamp; and
6. commits the claim.

The secrets-service request uses the dispatch ID as its idempotency key. If the
database commit is lost after remote registration, redelivery repeats the same
request and receives the same binding.

## Failure behavior

- Authentication, state, ownership, version, or binding conflicts fail closed.
- Secrets-service unavailability leaves the message unclaimed and eligible for
  bounded redelivery.
- An executor must not bootstrap, fetch inventory, or emit a started event until
  the claim succeeds.
- Terminal run processing cancels the binding using the stored dispatch ID.

Cancellation is reconciled by the scheduler after the consumer durably records
`successful`, `failed`, or `canceled`. A successful secrets-service response is
recorded in `credential_binding_canceled_at` with an immutable reason. If the
service is unavailable, that marker remains empty and the scheduler retries;
the secrets-service operation is idempotent for the same run and dispatch ID.

## Required schema

`execution_runs` gains:

- `dispatch_id uuid NOT NULL` for newly dispatched runs;
- `executor_identity text`;
- `credential_binding_created_at timestamptz`.

The executor identity becomes immutable once set. A database constraint or
trigger must reject replacement with a different identity.
